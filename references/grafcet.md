# GRAFCET 離散事件建模（MIAT 階段二）

## 為什麼是 GRAFCET

離散事件系統（Discrete-Event System, DES）＝狀態改變只發生在離散時間點的系統。每個 IDEF0 功能模組都視為一個獨立的 DES。

| 工具 | 不足 |
|---|---|
| 流程圖 Flow Chart | 缺乏階層化/結構化機制、缺乏狀態同步機制、缺乏平行/共時機制 |
| 狀態機 FSM | 缺乏階層化/模組化機制、缺乏狀態同步機制、缺乏平行/共時機制 |
| Petri Net | 不易結構化、模型過於複雜（限於學術） |

GRAFCET 由 Petri Net 簡化而來，1987 年成為國際標準，簡潔、嚴謹、能描述平行（concurrent）與分支（branching），廣用於 PLC 與自動化系統。理想 DES 建模工具四要求：能描述大量連續的離散狀態、能簡明表達同時發生、只描述因輸入改變而產生的行為、輸入/輸出行為清晰可見。

## 組成元件

- **狀態（step/state）**：單線正方形；**初始狀態**用雙線正方形（模型起點，如 main 函數）；方塊內黑圓點＝活躍（active），無點＝閒置。
- **動作（Action）**：與狀態相連的矩形，狀態活躍時執行；**雙線矩形＝子 GRAFCET**（動作本身是一個更詳細的 GRAFCET 模型）。
- **轉移條件（transition/condition）**：一條橫線，右側標註條件。
- **有向連結（directed connection）**：只能「狀態→轉移條件」或「轉移條件→狀態」。

## 語法規則（嚴格）

1. **交替規則**：狀態與轉移條件必須交替出現。兩個轉移之間必有狀態；兩個狀態之間必有轉移。
2. **每個轉移都要有明確條件**；即使自動轉移也要明寫 `=1`（永真）。
3. **禁止自迴圈**：狀態不畫箭頭指回自己。動作要持續執行就停留在該狀態，直到離開條件滿足。**若目前狀態的所有離開條件皆不成立，停留是合法且常用的等待語義**（例如分支條件未涵蓋的輸入組合，就停留原狀態）。
4. **一條轉移線不可連接到兩個不同的目標狀態**（除非是 AND 分歧雙線）。
5. 箭頭必須精確連接在轉移線上，不可畫在狀態方塊側面。
6. **平行架構（Concurrent）— 雙線**：
   - Divergence AND：條件滿足時，**所有**後續狀態同時激活。
   - Convergence AND：**所有**並行狀態都活躍且條件滿足，才同時轉移。
   - Divergence AND 與 Convergence AND 必須成對出現。
7. **分支架構（Branching）— 單線**：
   - Divergence OR：依先滿足的條件轉移到**唯一**一個後續狀態（if-then-else / case）。分支條件必須互斥（如 `C1` 與 `!C1`）。
   - Convergence OR：多個分支狀態匯合到共同狀態。
8. **同步語義**：條件滿足後，在下一個時脈正緣（clock positive edge）才發生轉移；來源狀態轉為非活躍、目標狀態轉為活躍。整體速度取決於最慢模組——效能瓶頸用平行/流水線（pipeline）解決。

## 階層式建模（Sub-GRAFCET）

- 複雜動作用雙線矩形表示子 GRAFCET。
- **編號規則**：主模型狀態 N 的子 GRAFCET，內部狀態從 N0 開始編號。狀態 2 → 20, 21, 22…；模組 A21 → 210, 211…；狀態 34 → 340, 341…。保證全系統狀態與變數命名唯一，避免團隊命名衝突。
- **父子握手（不需通訊協定）**：
  - 父狀態 XN 活躍 → 成為子控制器初始狀態的轉移條件（子從 N0 轉移到 N1）。
  - 子控制器最後的結束狀態（例如 X14）活躍 → 成為父控制器的轉移條件（`X1==1 && X14==1`）。
  - 子控制器結束狀態通常是 Null 動作，只作為轉移結束旗標。
- 建模完成後**必須附狀態/動作說明表**：每列＝狀態編號、動作名稱、函式名稱、功能說明。

## 建模範例

### 範例 1：沖壓機（順序迴圈）

三狀態迴圈：0 等待（初始）→ 條件 `m==1 && high_position` → 1 下降（Drop）→ 條件 `low_position` → 2 上升（Rise）→ 條件 `high_position` → 回 0。感測器讀數＝轉移條件；致動器動作＝狀態動作。

### 範例 2：化工攪拌器（Divergence/Convergence AND）

0 初始 → `sensor_OK` → 1（V1 Open）→ `L1` →【雙線 AND 分歧】同時激活 2（Start M 攪拌）與 3（V2 Open）→ `L2`【雙線 AND 匯聚】→ 4（Stop M、關閉閥門）→ 回 0。「同時打開閥門並啟動馬達」正是 FSM 無法描述、GRAFCET 擅長的。

### 範例 3：交通號誌（計時器模式）

0 等待 → `stby=1` → 1 RY（動作 `RY<='1'`、計時 `timeRY++`）→ `timeRY==5` → 2 GR（`timeGR++`）→ `timeGR==45` → 3 YR → `timeYR==5` → 4 RG（`timeRG++`）→ `timeRG==45` → 回 1。
**計時器模式**：進入狀態開始累加計時變數，轉移條件寫成計時到值。

### 範例 4：1001 序列辨識器（時序邏輯）

0 初始 → 輸入1 → 1 → 輸入0 → 2 → 輸入0 → 3 → 輸入1 → 4（辨識成功、輸出1）→ 回 1（繼續找下一個開頭）。

### 範例 5：Lidar 避障自走車（階層＋分支）

- **A0 主控制器**：0 → `start` → 1 感測 → `sensing_done` → 2 決策 → `decision_done` → 3 運動控制 → `action_done` → 回 0。
- **A1 感測**（狀態 10 起）：10 → 11 讀距離（`dist_ok`）→ 12 讀姿態（`imu_ok`）→ 回 10。
- **A2 決策**（狀態 20 起）：21 依 ToF 資料決策 →【OR 分歧】`safe`→forward／`obstacle_left`→turn_right／`obstacle_right`→turn_left／`blocked`→stop →【OR 匯聚】26 → `command_set` → 回 20。
- **A21 ToF-Lidar 掃描**（第三層，狀態 210 起）：210 → `start` → 211 控制雷達馬達角度 →【OR 分歧】212 記錄左側障礙物距離／213 記錄右側障礙物距離（每讀取週期「馬達角度++」）→ 掃描完成 → 214 雷達角度歸零、記錄左右障礙物數量 → 回 210。示範分支＋計數迴圈的第三層子模型。
- **A3 運動控制**（狀態 30 起）：31 →【OR 分歧】依指令執行 `car_forward()`/`car_turn_left()`/`car_turn_right()`/`car_stop()` →【OR 匯聚】36 → 回 30。

## 常見錯誤（課程明列）

| # | 錯誤 | 原因 |
|---|---|---|
| 1 | 兩個轉移條件之間沒有狀態 | 違反交替規則 |
| 2 | 兩個狀態之間沒有轉移條件 | 違反交替規則 |
| 3 | 一個轉移條件連接到兩個不同狀態 | 違反唯一路徑（該用 AND 分歧就用雙線） |
| 4 | 狀態轉移回自己（自迴圈） | 禁止；停留即可 |
| 5 | 箭頭畫在狀態方塊側面、未接在轉移線上 | 合成工具無法解析 |
| 6 | 一個狀態直接連出兩條轉移線 | 應為一條轉移線＋組合條件，或正規 OR 分歧 |

GRAFCET 不是流程圖：它是嚴謹的同步設計語言，語法錯誤會讓高階合成產生冗餘或錯誤的程式碼。

## 檢核清單

- [ ] 每個 IDEF0 葉模組一張 GRAFCET
- [ ] 初始狀態雙線框、編號符合階層規則（N → N0, N1…）
- [ ] 狀態/轉移交替、無自迴圈、永真寫 =1
- [ ] AND 雙線成對、OR 單線條件互斥
- [ ] 附狀態/動作/函式名稱/功能說明表
- [ ] 父子控制器以狀態暫存器銜接（XN 啟動子、子結束狀態當父條件）
